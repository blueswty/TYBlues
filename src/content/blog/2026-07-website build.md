## # 前言

这次开始认真折腾博客和周刊，出发点其实很简单：想做一点能长期留存、也真正属于自己的东西。

某种程度上，这也算是受到了 [tw93 的潮流周刊](https://weekly.tw93.fun/) 的影响。看久了会发现，持续记录本身就是一件很有力量的事。与其总想着“等准备好了再开始”，不如先把第一篇写出来。

## # 开始用上 Codex

最近 `Codex` 的讨论度很高，各种自动完成、自动修改、自动执行的能力确实让人很难不关注。此前我更多还是在用豆包和 `Trae`，坦白说，`Trae` 也并不差，帮我完成过一些 demo 工程，只是总觉得离“真正顺手的生产力工具”还差一点意思。

于是，这周我开始找渠道，把 `Codex` 用起来。

国内使用 `OpenAI` 相关服务多少还是要绕一些路。没有 `Visa` 卡，再加上我是鸿蒙手机用户，也没法像苹果生态用户那样直接购买 `ChatGPT`。后来同事推荐了一个 `Codex API` 服务，对我这种主要拿它来处理代码、文档和文件流的场景来说，已经相当够用了。

[Codex API 服务 - 国内直连 OpenAI](https://api.codexcn.com/)

## # AI 的进步，快得有点夸张

这段时间我拿 AI 试了几个全新的工程需求，感受非常直接：它的发展速度已经远远超出了我的预期。和半年前相比，不只是“更聪明了一点”，而是已经出现了非常明显的代差。

就拿嵌入式开发来说，现阶段一个质量不错的 `STM32` 工程，很多时候真的可以在几乎不手写代码的前提下推进起来。当然，这背后也有前提：需求描述要足够明确，上下文准备要足够完整，约束条件也要提前说明。AI 并不是替你省掉思考，而是把“实现”这部分的门槛和体力活大幅压低了。

也正是因为这种信心加持，我才顺手把“注册域名、选建站方案、落地部署”这一整套事情都推进了下去。

## # 建站方案怎么选

在开始动手之前，我先让 AI 帮我从长期维护成本、扩展性和上手效率几个角度做了一轮方案筛选。最后给出的建议是：

> **最终建议**
>
> 如果目标是“长期省心，同时保留足够高的功能上限”，那么更适合我的方案是：
>
> - 首选：`Astro + Markdown/MDX + GitHub + Cloudflare Pages + R2`
> - 备选：`Hugo + GitHub + Cloudflare Pages`
> - 暂不建议：一开始就上 `Next.js + 数据库 + CMS`
>
> 原因也很现实：后者虽然更重，但对当前阶段来说，维护成本、复杂度和故障面都会明显变大。

![image-20260714155831782](https://picgocloud.com/m/ae45605a-008a-4cf8-acda-507ca8c711c1.png)

事实证明，这个判断是对的。博客建成之后，通过 `Cloudflare Pages` 挂载，再绑定域名访问，整个链路很顺。后续只要把内容按 `Markdown` 的形式提交到 `GitHub`，站点就可以自动更新，博客和周刊都能无缝发布。

这种工作流最让我满意的一点是：内容生产和网站维护被很好地拆开了。写作时只管写，发布时只管提交，剩下的交给自动化。

## # 这套技术栈的使用感受

这一周把站搭起来之后，也顺手把几块关键工具链串通了，整体体验比预期更舒服。

- `Astro` 很适合这种以内容为主的博客场景。页面结构清晰，内容组织也自然，既能保留静态站点的轻量，也给后续加功能留出了空间。
- `Markdown` 依旧是我最喜欢的内容输入方式。它足够纯粹，不会把注意力从“写什么”拉偏到“怎么排版”上。
- `GitHub + Cloudflare Pages` 这套组合很省心。前者负责版本管理和内容托管，后者负责自动构建和发布，稳定、直接，而且几乎不用额外操心运维细节。
- 如果后面相册、附件或者资源文件继续增加，`R2` 这种对象存储方案也很值得接进来，适合把静态资源进一步拆分和优化。

## # 写作、配图和评论系统

`Markdown` 编辑器我还是选了 `Typora`。它对我来说最大的好处不是“功能多”，而是足够顺手。很多年前朋友就推荐过，从免费使用到付费...

![image-20260714153956695](https://picgocloud.com/m/92751bc9-7161-4f5d-94db-1bd143c0f93f.png)

配图方面，我在 `Typora` 里配置了自动上传图片到图床，目前用的是 `PicGo Cloud`。写作时插图、上传、回链可以一气呵成，省掉了很多手动搬运的步骤。现阶段它提供的免费额度对个人博客来说也基本够用。

![image-20260714154051300](https://picgocloud.com/m/bdd01bcf-495f-42ca-8af1-e4bae41f9d13.png)

另外，相册里有些图片是相机原图，单张体积可能超过 `8MB`。这种图如果不处理，页面加载速度会明显变慢，所以我额外用了 `Recompressor` 做压缩优化。它支持多种压缩方案，对比结果也直观，而且全免费、无广告，体验相当不错。

![image-20260714163947208](https://picgocloud.com/m/3197c08e-72af-4104-85ac-aae80b95bcca.png)

博客和相册都有了，那评论区自然也不能少。我最后接入的是 `giscus`，基于 `GitHub` 账号登录，配置过程比较顺，风格上也和静态博客非常契合。
